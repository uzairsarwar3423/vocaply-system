1. API Standards & Conventions
Request Headers
Authorization: Bearer <access_token>      ← Required on all protected routes
Content-Type:  application/json           ← Always
Accept:        application/json           ← Always
Standard Success Response
json{
  "success": true,
  "data": { },
  "meta": {
    "page": 1,
    "limit": 20,
    "total": 150,
    "hasMore": true
  }
}
Standard Error Response
json{
  "success": false,
  "error": {
    "code": "NOT_FOUND",
    "message": "Meeting not found",
    "details": {
      "field": "meetingId",
      "value": "mtg_invalid"
    }
  }
}
HTTP Status Codes Used
200  OK              → Success (GET, PATCH, DELETE)
201  Created         → Resource created (POST)
400  Bad Request     → Malformed request body
401  Unauthorized    → Missing or invalid token
402  Payment Req     → Plan limit reached
403  Forbidden       → Valid token but insufficient role
404  Not Found       → Resource not found
409  Conflict        → Duplicate resource
422  Unprocessable   → Validation failed (field errors)
429  Too Many Req    → Rate limit exceeded
500  Server Error    → Unexpected error
502  Bad Gateway     → Third-party integration failed
Error Codes
AUTH_REQUIRED        401  → No token provided
TOKEN_EXPIRED        401  → Access token expired
TOKEN_INVALID        401  → Malformed or tampered token
FORBIDDEN            403  → Valid token, wrong role
NOT_FOUND            404  → Resource does not exist
VALIDATION_ERROR     422  → Invalid input fields
DUPLICATE            409  → Already exists
PLAN_LIMIT           402  → Exceeded plan quota
ACCOUNT_LOCKED       429  → Too many failed login attempts
RATE_LIMITED         429  → API rate limit hit
INTEGRATION_ERROR    502  → Jira, Slack, etc. API failed
INTERNAL_ERROR       500  → Unexpected server error
Pagination Query Params
?page=1              → Page number (default: 1, min: 1)
?limit=20            → Items per page (default: 20, max: 100)
?sortBy=createdAt    → Sort field
?sortOrder=desc      → Sort direction: asc | desc
Date Format
All dates:   ISO 8601 → "2026-05-12T09:00:00Z"
Date only:   "2026-05-12"
All times:   UTC timezone

2. Auth APIs
POST /auth/register
Register new user with email and password.
Auth:   Public (no token required)

Request Body:
{
  "name":     string   REQUIRED  min:2  max:100
              → Full name of the user
              → Example: "Ali Raza"

  "email":    string   REQUIRED
              → Valid email address, will be lowercased
              → Must be unique in the system
              → Example: "ali@techflow.com"

  "password": string   REQUIRED  min:8  max:128
              → Must contain: uppercase letter, number, special char
              → Example: "MyPass@123"
}

Success Response 201:
{
  "success": true,
  "data": {
    "message": "Registration successful. Check your email to verify your account.",
    "email": "ali@techflow.com"
  }
}

Error Responses:
  409 DUPLICATE       → Email already registered
  422 VALIDATION_ERROR → Invalid email, weak password
    {
      "error": {
        "code": "VALIDATION_ERROR",
        "message": "Validation failed",
        "details": {
          "fields": {
            "password": "Password must contain at least one uppercase letter",
            "email": "Invalid email format"
          }
        }
      }
    }

POST /auth/login
Login with email and password.
Auth:   Public

Request Body:
{
  "email":    string   REQUIRED
              → Registered email address

  "password": string   REQUIRED
              → Account password
}

Success Response 200:
{
  "success": true,
  "data": {
    "accessToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "user": {
      "id":        "usr_clx01abc",
      "name":      "Ali Raza",
      "email":     "ali@techflow.com",
      "avatarUrl": "https://...",
      "timezone":  "Asia/Karachi",
      "role":      "MANAGER",
      "team": {
        "id":   "team_clx02xyz",
        "name": "TechFlow Engineering",
        "slug": "techflow-eng",
        "plan": "GROWTH"
      },
      "createdAt": "2026-01-15T09:00:00Z"
    }
  }
}

Side Effect:
  → HttpOnly cookie set: vocaply_refresh=<refresh_token>
    Secure: true, SameSite: Strict, Path: /auth/refresh
    Expires: 30 days from now

Error Responses:
  401 → Invalid email or password (generic, never reveal which)
  403 → Email not verified
    { "error": { "code": "EMAIL_NOT_VERIFIED", "message": "Please verify your email" } }
  429 → Account locked (too many failed attempts)
    { "error": { "code": "ACCOUNT_LOCKED", "message": "Account locked for 15 minutes" } }

POST /auth/logout
Logout and invalidate refresh token.
Auth:   Required

Request Body: none

Success Response 200:
{
  "success": true,
  "data": {
    "message": "Logged out successfully"
  }
}

Side Effect:
  → Refresh token deleted from database
  → HttpOnly cookie cleared

Error Responses:
  401 → No valid session found

POST /auth/refresh
Get new access token using refresh token from cookie.
Auth:   Public (uses HttpOnly cookie)

Request Body: none

Cookies required:
  vocaply_refresh=<refresh_token>   ← Sent automatically by browser

Success Response 200:
{
  "success": true,
  "data": {
    "accessToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
  }
}

Side Effect:
  → Old refresh token deleted from DB
  → New refresh token issued → new HttpOnly cookie set
  → Token rotation happens automatically

Error Responses:
  401 → Refresh token missing, invalid, or expired
        → Frontend must redirect to /login

GET /auth/verify-email
Verify email with token from email link.
Auth:   Public

Query Parameters:
  token   string   REQUIRED
          → The verification token from the email link
          → Example: ?token=abc123def456...

Success Response 200:
{
  "success": true,
  "data": {
    "accessToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "user": { ...user object with team },
    "message": "Email verified successfully"
  }
}

Side Effect:
  → user.emailVerified set to true
  → Verification token deleted (one-time use)
  → New access + refresh tokens issued (auto-login)

Error Responses:
  400 → Token invalid or already used
  410 → Token expired (resend available)

POST /auth/resend-verification
Resend verification email.
Auth:   Public

Request Body:
{
  "email": string   REQUIRED
           → The email to resend verification to
}

Success Response 200:
{
  "success": true,
  "data": {
    "message": "Verification email sent if account exists"
  }
}

Note: Always returns 200 even if email not found (security)
Rate limit: 1 resend per 60 seconds per email

POST /auth/forgot-password
Request a password reset email.
Auth:   Public

Request Body:
{
  "email": string   REQUIRED
           → Email address of the account
}

Success Response 200:
{
  "success": true,
  "data": {
    "message": "If that email exists, a reset link has been sent"
  }
}

Note: Always returns 200 even if email not found (security)
Rate limit: 3 requests per hour per email

POST /auth/reset-password
Set new password using reset token.
Auth:   Public

Request Body:
{
  "token":       string   REQUIRED
                 → Reset token from the email link

  "newPassword": string   REQUIRED  min:8  max:128
                 → New password (same strength rules as register)
}

Success Response 200:
{
  "success": true,
  "data": {
    "accessToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "user": { ...user object },
    "message": "Password reset successful"
  }
}

Side Effect:
  → password_hash updated in DB
  → Reset token marked as used (one-time)
  → ALL existing refresh tokens for this user invalidated
  → New session issued (auto-login)

Error Responses:
  400 → Token invalid or already used
  410 → Token expired (request new reset)
  422 → Password too weak

GET /auth/google
Initiate Google OAuth flow.
Auth:   Public

Query Parameters: none

Response 302 (Redirect):
  → Redirects browser to Google OAuth consent screen:
    https://accounts.google.com/o/oauth2/v2/auth
      ?client_id=GOOGLE_CLIENT_ID
      &redirect_uri=https://api.vocaply.com/auth/google/callback
      &response_type=code
      &scope=openid email profile
      &state=<32-byte-random-state>
      &access_type=offline

GET /auth/google/callback
Google OAuth callback (called by Google, not frontend).
Auth:   Public (verified via state parameter)

Query Parameters (sent by Google):
  code    string   REQUIRED
          → Authorization code from Google

  state   string   REQUIRED
          → Must match state stored in Redis (CSRF check)

  error   string   optional
          → Set if user denied access: "access_denied"

Success Response 302 (Redirect):
  → https://vocaply.com/dashboard
  → HttpOnly cookie set with refresh token
  → Access token set in URL fragment (frontend reads once)
  OR
  → https://vocaply.com/onboarding (if new user)

Error Response 302 (Redirect):
  → https://vocaply.com/login?error=oauth_failed

GET /auth/github
Initiate GitHub OAuth flow.
Auth:   Public

Response 302 (Redirect):
  → Redirects to GitHub OAuth consent screen:
    https://github.com/login/oauth/authorize
      ?client_id=GITHUB_CLIENT_ID
      &redirect_uri=https://api.vocaply.com/auth/github/callback
      &scope=read:user user:email
      &state=<32-byte-random-state>

GET /auth/github/callback
GitHub OAuth callback.
Auth:   Public (verified via state parameter)

Query Parameters (sent by GitHub):
  code    string   REQUIRED
  state   string   REQUIRED

Success Response 302 → /dashboard or /onboarding
Error Response 302   → /login?error=oauth_failed

4. Meeting APIs
POST /meetings
Create a meeting and auto-schedule the bot.
Auth:   Required

Request Body:
{
  "title": string   REQUIRED   min:1   max:255
           → Meeting title/name
           → Example: "Monday Engineering Standup"

  "platform": string   REQUIRED
              → Meeting platform
              → Allowed: "ZOOM" | "GOOGLE_MEET" | "TEAMS" | "WEBEX" | "MANUAL"

  "meetingUrl": string   REQUIRED   must be valid URL
                → Full meeting join URL
                → Example: "https://zoom.us/j/123456789"
                → Vocaply extracts platform ID for deduplication

  "scheduledAt": string   REQUIRED   ISO 8601 datetime
                 → When the meeting is scheduled to start
                 → Example: "2026-05-12T09:00:00Z"
                 → Must be in the future (within 7 days for bot scheduling)

  "calendarEventId": string   optional
                     → Google Calendar event ID (used for deduplication)
                     → Example: "gcal_event_abc123"
}

Success Response 201:
{
  "success": true,
  "data": {
    "id":          "mtg_clx05pqr",
    "title":       "Monday Engineering Standup",
    "platform":    "ZOOM",
    "meetingUrl":  "https://zoom.us/j/123456789",
    "recallBotId": "bot_recall_abc123",
    "status":      "SCHEDULED",
    "scheduledAt": "2026-05-12T09:00:00Z",
    "createdAt":   "2026-05-11T10:00:00Z",
    "message":     "Bot scheduled — will join 2 minutes before meeting start"
  }
}

Error Responses:
  402 PLAN_LIMIT → Monthly meeting limit reached
    {
      "error": {
        "code": "PLAN_LIMIT",
        "message": "Monthly meeting limit reached (5/5 on Free plan)",
        "details": {
          "used":    5,
          "limit":   5,
          "plan":    "FREE",
          "upgradeUrl": "https://vocaply.com/settings/billing"

PATCH /commitments/:commitmentId/status
Update commitment status.
Auth:   Required
        → MEMBER: can only update their OWN commitments
        → MANAGER+: can update any team member's commitments

Path Parameters:
  commitmentId   string   REQUIRED

Request Body:
{
  "status": string   REQUIRED
            → New status to set
            → Allowed: "FULFILLED" | "DEFERRED" | "CANCELLED"
            → Cannot set to "PENDING" or "MISSED" (system-managed)

  "note": string   optional   max:500
          → Reason or context for the status change
          → Example: "Pushed to next sprint due to dependency"

  "newDueDate": string   conditional   ISO 8601 datetime
                → REQUIRED when status = "DEFERRED"
                → Must be in the future
                → Example: "2026-05-20T23:59:59Z"
}

Success Response 200:
{
  "success": true,
  "data": {
    "id":         "com_01",
    "status":     "FULFILLED",
    "resolvedAt": "2026-05-12T10:15:00Z",
    "updatedAt":  "2026-05-12T10:15:00Z"
  }
}

Side Effects:
  → FULFILLED: Slack notification to team, score updated, Jira/Linear updated
  → DEFERRED:  Reminder rescheduled to new due date
  → CANCELLED: Removed from open commitments, no further alerts

Error Responses:
  403 → MEMBER trying to update another member's commitment
  404 → Commitment not found
  422 → Invalid status transition
  422 → newDueDate required for DEFERRED status
  422 → newDueDate must be in the future

11. Webhook APIs

Note: These endpoints are called by third-party services, not frontend.
They use signature verification instead of JWT auth.

POST /webhooks/recall
Receive events from Recall.ai meeting bot.
Auth:   Signature verification only
        Header: X-Recall-Signature: sha256=<HMAC-SHA256>

Events handled:

EVENT: bot.joining_call
{
  "event": "bot.joining_call",
  "data": {
    "bot_id":      "bot_recall_abc123",
    "meeting_url": "https://zoom.us/j/123456789"
  }
}
→ Update meeting status: BOT_JOINING

EVENT: bot.in_waiting_room
{
  "event": "bot.in_waiting_room",
  "data": { "bot_id": "bot_recall_abc123" }
}
→ No status change (waiting for host to admit)

EVENT: bot.recording_started
{
  "event": "bot.recording_started",
  "data": { "bot_id": "bot_recall_abc123" }
}
→ Update meeting status: RECORDING, set startedAt

EVENT: bot.done
{
  "event": "bot.done",
  "data": {
    "bot_id":     "bot_recall_abc123",
    "meeting_url":"https://zoom.us/j/123456789",
    "transcript": [
      {
        "speaker":     "Speaker 1",
        "is_final":    true,
        "words": [
          { "text": "Good", "start_time": 5.1, "end_time": 5.4, "confidence": 0.98 }
        ]
      }
    ],
    "output_video": {
      "status":      "done",
      "data":        { "download_url": "https://..." }
    }
  }
}
→ Store transcript in MongoDB
→ Push job to transcribe.queue
→ Update meeting: endedAt, status: PROCESSING

EVENT: bot.failed
{
  "event": "bot.failed",
  "data": {
    "bot_id": "bot_recall_abc123",
    "reason": "Meeting ended before bot could join"
  }
}
→ Update meeting status: FAILED
→ Alert team via Slack/email

Response 200 (always):
{ "received": true }

Response 400:
{ "received": false, "error": "Invalid signature" }

POST /webhooks/stripe
Receive billing events from Stripe.
Auth:   Signature verification only
        Header: Stripe-Signature: t=1715000000,v1=<HMAC>

Events handled:

EVENT: checkout.session.completed
→ First-time purchase: activate subscription, update team plan

EVENT: customer.subscription.created
{
  "type": "customer.subscription.created",
  "data": {
    "object": {
      "id":         "sub_abc123",
      "customer":   "cus_xyz789",

      "plan": { "id": "price_growth_monthly" },
      "current_period_start": 1715000000,
      "current_period_end":   1717678400
    }
  }
}
→ Update team plan in DB
→ Reset monthly usage counters

EVENT: customer.subscription.updated
→ Plan changed (upgrade/downgrade): update DB

EVENT: customer.subscription.deleted
→ Subscription cancelled: downgrade team to FREE plan
→ Send cancellation email to admin

EVENT: invoice.payment_succeeded
→ Send receipt email to admin
→ Update subscription period dates

EVENT: invoice.payment_failed
{
  "type": "invoice.payment_failed",
  "data": {
    "object": {
      "customer":       "cus_xyz789",
      "amount_due":     7900,
      "attempt_count":  1,
      "next_payment_attempt": 1715259600
    }
  }
}
→ Send payment failed warning email
→ If attempt_count >= 3 → suspend account

Response 200 (always):
{ "received": true }

POST /webhooks/jira
Receive issue update events from Jira (reverse sync).
Auth:   Jira webhook secret verification

Events handled:

EVENT: jira:issue_updated
{
  "webhookEvent":  "jira:issue_updated",
  "issue": {
    "id":     "10042",
    "key":    "TECH-142",
    "fields": {
      "status":     { "name": "Done" },
      "assignee":   { "emailAddress": "ahmed@techflow.com" },
      "summary":    "Fix payment bug in checkout"
    }

}
→ Find action_item where jiraIssueId = "TECH-142"
→ If status = "Done" → mark action_item.completed = true
→ Check if action item linked to commitment
→ If yes → mark commitment FULFILLED
→ Notify team via Slack + Socket.io

Response 200:
{ "received": true }
